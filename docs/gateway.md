# Gateway — technical documentation

The gateway is the single entry point for all client requests. Nothing in the frontend talks directly to a database or microservice — every request goes through here first. This document explains how it is structured, what each part does, and how the pieces fit together.

---

## What the gateway does

At a high level, the gateway has four responsibilities:

1. **Routing** — receives HTTP requests from the frontend and directs them to the right handler
2. **Authentication** — validates JWT tokens on protected routes before the handler runs
3. **Data access** — reads and writes to the local JSON stores (swappable for Supabase later)
4. **Proxying** — forwards requests to the RAG and CV microservices (to be implemented)

It is written in Go using the Gin web framework. The entry point is `cmd/main.go`, which wires everything together on startup.

## Route groups

### Products — `/api/products`

Public routes. No authentication required.

| Method | Path                   | Description                                  |
| ------ | ---------------------- | -------------------------------------------- |
| GET    | `/api/products`        | Paginated product list with optional filters |
| GET    | `/api/products/groups` | Full taxonomy of groups and types            |
| GET    | `/api/products/:id`    | Single product with full attributes          |

**Filtering** is done via query parameters. All are optional and combinable:

- `group` — filters by top-level group: `structural`, `guidance`, `payload`, `propulsion`
- `type` — filters by product type: `rocket_frame`, `liquid_engine`, etc.
- `search` — case-insensitive name search
- `page` — page number, defaults to 1
- `limit` — items per page, defaults to 20, max 100

Example request:

```
GET /api/products?group=propulsion&search=merlin&page=1&limit=10
```

**Product attributes** are stored as a JSONB blob (`attributes` field) rather than as fixed columns. This allows each of the 13 product types to have completely different fields without requiring separate database tables. When you fetch a single product, the full `attributes` object is returned. When listing products, attributes are omitted to keep responses fast.

The typed attribute structs (e.g. `LiquidEngineAttributes`, `NoseConeAttributes`) live in `models/product.go` and are used server-side when any logic needs to inspect specific fields.

---

### Auth — `/api/auth`

Mixed visibility. Register, login, and logout are public. The `/me` endpoint requires a valid JWT.

| Method | Path                 | Auth | Description                       |
| ------ | -------------------- | ---- | --------------------------------- |
| POST   | `/api/auth/register` | No   | Create account, returns JWT       |
| POST   | `/api/auth/login`    | No   | Validate credentials, returns JWT |
| POST   | `/api/auth/logout`   | No   | Stateless — client discards token |
| GET    | `/api/auth/me`       | Yes  | Returns current user from JWT     |

Passwords are never stored in plain text. They are hashed using bcrypt before being written to `users.json`. The hash is never returned in any response — the `UserPublic` struct intentionally excludes it.

On successful register or login, an `AuthResponse` is returned containing the JWT token and a `UserPublic` object. The frontend should store the token (e.g. in localStorage) and include it as a `Bearer` token in the `Authorization` header on all subsequent protected requests.

---

### Orders — `/api/orders`

All routes require authentication. Users can only access their own orders.

| Method | Path                     | Description                             |
| ------ | ------------------------ | --------------------------------------- |
| GET    | `/api/orders`            | Paginated list of current user's orders |
| POST   | `/api/orders`            | Create a new order                      |
| GET    | `/api/orders/:id`        | Single order by ID                      |
| PUT    | `/api/orders/:id/cancel` | Cancel a pending order                  |

**Creating an order** requires a list of items (product ID + quantity) and a shipping address. The gateway looks up each product, checks it is in stock, calculates line totals and a flat-rate shipping cost, and persists the order with status `pending`.

**Cancelling an order** only succeeds if the order is in a cancellable status. The cancellable statuses are:

```
pending → can cancel
payment_processing → can cancel
payment_failed → can cancel
confirmed → can cancel
preparing → cannot cancel
shipped → cannot cancel
in_transit → cannot cancel
delivered → cannot cancel
cancelled → cannot cancel (already cancelled)
refund_pending → cannot cancel
refunded → cannot cancel
```

If a user tries to access or cancel an order that belongs to another user, the gateway returns `403 Forbidden` — not `404`. This is intentional: returning 404 would leak information about whether an order ID exists at all.

## Models

### Product

```go
type Product struct {
    ID         string          // unique identifier e.g. "le-001"
    Name       string          // display name e.g. "Merlin 1D Vacuum"
    Group      ProductGroup    // structural | guidance | payload | propulsion
    Type       ProductType     // rocket_frame | liquid_engine | nose_cone | etc.
    Price      float64         // price in USD
    ImageURL   string          // path to product image
    InStock    bool            // whether the product is available to order
    StockCount int             // number of units available
    Attributes json.RawMessage // JSONB blob — shape varies by product type
    CreatedAt  string          // RFC3339 timestamp
    UpdatedAt  string          // RFC3339 timestamp
}
```

The `Attributes` field contains different fields depending on the product type. For example:

```json
// liquid_engine attributes
{
  "max_thrust_kn": 934,
  "specific_impulse_s": 348,
  "burn_time_s": 397,
  "chamber_pressure_bar": 97,
  "weight_kg": 470,
  "gimbal_range_deg": 5.0
}

// nose_cone attributes
{
  "material": "PICA-X Ablative Composite",
  "length_m": 3.9,
  "base_diameter_m": 3.66,
  "mass_kg": 840,
  "thermal_resistance": "1650°C sustained",
  "payload_compatibility": "Dragon capsule, custom payloads up to 3.5m dia",
  "reusability_cycles": 10
}
```

All 13 typed attribute structs are defined in `models/product.go` and follow the same naming conventions.

### User

```go
// Full record — only ever used internally, never sent to the client
type User struct {
    ID             string  // UUID
    Email          string
    Name           string
    HashedPassword string  // bcrypt hash — never exposed
    CreatedAt      string  // RFC3339 timestamp
}

// Safe version — returned in all API responses
type UserPublic struct {
    ID        string
    Email     string
    Name      string
    CreatedAt string
}
```

### Order

```go
type Order struct {
    ID              string          // UUID
    UserID          string          // UUID of the owning user
    Status          OrderStatus     // see status lifecycle below
    Items           []OrderItem     // line items
    ShippingAddress ShippingAddress // delivery destination
    Subtotal        float64         // sum of all line totals
    ShippingCost    float64         // flat rate shipping fee
    Total           float64         // subtotal + shipping cost
    Notes           string          // optional order notes
    CreatedAt       string          // RFC3339 timestamp
    UpdatedAt       string          // RFC3339 timestamp
}

type OrderItem struct {
    ProductID   string   // references a product ID
    ProductName string   // denormalised for display — survives product changes
    ProductType string   // denormalised for display
    Quantity    int
    UnitPrice   float64
    LineTotal   float64  // UnitPrice * Quantity
}
```

Product name and type are denormalised into the order item intentionally. If a product is renamed or removed in the catalog later, the order history still shows what was actually purchased at the time.

The full order status lifecycle:

```
pending
  → payment_processing
      → payment_failed → (can retry)
      → confirmed
          → preparing
              → shipped
                  → in_transit
                      → delivered
                          → refund_pending
                              → refunded
  → cancelled  (from: pending, payment_processing, payment_failed, confirmed)
```

## How authentication works

### Overview

Starbound uses **JSON Web Tokens (JWT)** for authentication. Rather than storing sessions on the server, the server issues a signed token when a user logs in. The client stores this token and sends it with every subsequent request. The server validates the token's signature on each request — no database lookup needed.

### Registration flow

```
Client                          Gateway
  |                               |
  | POST /api/auth/register       |
  | { email, name, password }     |
  |------------------------------>|
  |                               | 1. Validate request fields
  |                               | 2. Check email not already taken
  |                               | 3. Hash password with bcrypt
  |                               | 4. Generate UUID for user ID
  |                               | 5. Save user to users.json
  |                               | 6. Generate signed JWT (24hr expiry)
  |                               |
  | { token, user }               |
  |<------------------------------|
  |                               |
  | Store token in localStorage   |
```

### Login flow

```
Client                          Gateway
  |                               |
  | POST /api/auth/login          |
  | { email, password }           |
  |------------------------------>|
  |                               | 1. Look up user by email
  |                               | 2. Compare password with bcrypt hash
  |                               | 3. If mismatch → 401 (same error for
  |                               |    unknown email — don't reveal existence)
  |                               | 4. Generate signed JWT (24hr expiry)
  |                               |
  | { token, user }               |
  |<------------------------------|
```

### Protected request flow

```
Client                          Gateway
  |                               |
  | GET /api/orders               |
  | Authorization: Bearer <token> |
  |------------------------------>|
  |                               | RequireAuth middleware:
  |                               | 1. Extract token from header
  |                               | 2. Verify signature with JWT_SECRET
  |                               | 3. Check token not expired
  |                               | 4. Store user_id, email, name in
  |                               |    Gin context for handler to use
  |                               |
  |                               | Handler runs:
  |                               | userID := c.GetString("user_id")
  |                               | → fetch orders for this user
  |                               |
  | { data: [...orders] }         |
  |<------------------------------|
```

### JWT structure

The token payload (claims) contains:

```json
{
  "user_id": "246667...",
  "email": "test@starbound.com",
  "name": "Test User",
  "iss": "starbound",
  "exp": 1773578846,
  "iat": 1773492446
}
```

The token is signed with `HS256` using the `JWT_SECRET` environment variable. In development, if `JWT_SECRET` is not set, a hardcoded fallback is used — this must be changed before any kind of production deployment.

### Logout

JWT is stateless — the server holds no session to invalidate. Logout is handled entirely client-side by deleting the stored token. The `POST /api/auth/logout` endpoint exists as a consistent API contract for the frontend to call, but it does nothing server-side.

For a production system you would implement a token blacklist (e.g. store revoked token IDs in Redis with a TTL matching the token expiry). That is out of scope for this project.

## Data storage

Currently all data is persisted in local JSON files:

| File                             | Contents                              |
| -------------------------------- | ------------------------------------- |
| `internal/db/products_seed.json` | Product catalog — read-only seed data |
| `internal/db/users.json`         | Registered user accounts              |
| `internal/db/orders.json`        | Placed orders                         |

The product database is accessed through the `DB` interface, which means swapping the implementation to a real Supabase client later only requires implementing that interface — none of the handler code needs to change.

The user and order stores are concrete structs (`UserStore`, `OrderStore`) that will be refactored into a similar interface pattern when Supabase is integrated.

All stores use a `sync.RWMutex` to prevent race conditions when multiple requests read and write concurrently.

## Test suites

Tests live in `gateway/tests/` and use the `testify` library for assertions. Each test file mirrors a route group.

### Running tests

```bash
cd gateway
go test ./tests/... -v
```

### handlers_test.go — product endpoint tests (12 tests)

Covers the product listing, filtering, search, pagination, single product lookup, attribute validation, and group taxonomy endpoint. Uses `NewTestDB()` which points at the real seed file — products are read-only so there is no risk of test pollution.

Key scenarios covered:

- Response shape matches the `ProductListResponse` struct
- Group and type filters return only matching products
- Search is case-insensitive
- Pagination respects page and limit parameters
- A search with no matches returns an empty list, not an error
- A single product response includes the full attributes blob
- Fetching a non-existent ID returns 404
- The groups endpoint always returns exactly four groups

### auth_test.go — auth endpoint tests (16 tests)

Covers the full registration and login lifecycle, input validation, duplicate detection, password security, and JWT-protected routes. Uses `NewTestUserStore()` which creates a temporary JSON file for each test run — tests never touch the real `users.json`.

Key scenarios covered:

- Successful registration returns a token and a user object
- The hashed password never appears anywhere in the response
- Duplicate email registration returns 409
- Short passwords, invalid emails, and short names are rejected at 400
- Login with wrong password returns 401 with the same error as unknown email
- The `/me` endpoint returns the correct user for a valid token
- Missing, invalid, and malformed Authorization headers all return 401

### orders_test.go — order endpoint tests (20 tests)

Covers the full order lifecycle from creation through cancellation, including ownership enforcement and status transition rules. Uses `NewTestOrderStore()` which creates a temporary JSON file per test run.

Key scenarios covered:

- Order creation calculates subtotal, shipping, and total correctly
- Product name and type are denormalised into line items
- Ordering an out-of-stock or non-existent product returns 422
- Missing items or shipping address returns 400
- A new user's order list is empty
- Order list only returns orders belonging to the authenticated user
- Two different users cannot access each other's orders (403, not 404)
- An order can be cancelled from a cancellable status
- Cancelling an already-cancelled order returns 400
- All order endpoints return 401 with no token

## Swagger / OpenAPI

The gateway self-documents via Swaggo. Annotations in handler comments are compiled into an OpenAPI spec by running:

```bash
swag init -g cmd/main.go --output docs
```

The generated `docs/` folder is committed to the repo. The live interactive UI is served at:

```
http://localhost:8000/swagger/index.html
```

When adding new endpoints, always add the corresponding Swaggo annotations to the handler function and re-run `swag init` to keep the spec in sync.
